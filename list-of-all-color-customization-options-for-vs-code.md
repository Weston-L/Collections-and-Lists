<H1>LIST OF ALL COLOUR CUSTOMISATION OPTIONS YOU CAN PUT IN YOUR SETTINGS.JSON TO CHANGE COLOURS AND STYLE IN YOUR CODE</HI>
<H2> - FOR VS CODE - </H2>
Here's a fairly comprehensive "blank palette" that combines the common built-in categories, a large set of semantic token types, and many frequently-used TextMate scopes. 

You can paste it into your `settings.json` and fill in colours/styles as you experiment.


Supported `fontStyle` values include combinations of:

italic, bold, underline, strikethrough, or an empty string.

------------------------------------------------
A useful trick: assign **wildly different colours** at first (bright red, lime green, cyan, yellow, magenta, etc.). That makes it immediately obvious what each token category affects. Once you've identified what everything controls, you can replace the "testing colours" with a polished palette.
----------------

A few fun things to try:

        * Make **parameters** italic so function signatures are easier to scan.
        * Give **constants** a bold style to make them stand out from variables.
        * Color **class names** and **function names** very differently so code structure pops visually.
        * Use a subtle background on **Markdown inline code** or **regular expressions**.
        * Give **deprecated code** (if your language server supports it) a strikethrough via semantic tokens.

        And if you ever find yourself wondering, "Why didn't that colour change anything?", it's usually one of three reasons:

        1. The theme is using **semantic highlighting** instead of TextMate scopes.
        2. The language extension uses a different scope name than expected.
        3. The token isn't exposed by that language grammar.

        When that happens, **Developer: Inspect Editor Tokens and Scopes** is your best friend.

Have fun building your own editor aesthetyc!
